# F1 数据可视化微信小程序 — 产品设计文档

---

## 一、产品定位

面向 F1 普通观众和硬核车迷，提供赛后数据可视化分析 + AI 专业解读。
用交互图表替代静态截图，让遥测数据「动起来」；用 AI 把数字翻译成人话，降低硬核数据的阅读门槛。

---

## 二、MVP 范围（第一版）

> 先做历史赛事数据，实时功能在 v2 迭代。

### 2.1 支持的数据维度

| 维度 | 内容 |
|------|------|
| 遥测对比 | 两车速度 / 油门 / 刹车 / 档位，横轴对齐弯角编号 |
| 圈时分析 | 单车多圈趋势、轮胎策略叠加 |
| 排位赛结果 | Q1 / Q2 / Q3 各车手最快圈排名 |
| 积分榜 | 车手 & 车队当前积分走势 |
| **AI 分析报告** | **基于数据自动生成多维度专业解读** |

### 2.2 不做（MVP 阶段排除）

- 实时数据推送
- 用户登录 / 收藏
- 社区评论
- 付费功能

│  数据格式:  JSON（供前端 ECharts 直接消费）   │
│  AI 分析:   规则引擎 → LLM API (Claude/GPT)  │
└─────────────────┬───────────────────────────┘
                  │
       ┌──────────┴──────────┐
       ▼                     ▼
┌─────────────┐      ┌──────────────────┐

└─────────────┘

**部署方案（优先级排序，待选）**

1. 轻量云服务器（2核4G）— 最灵活，FastF1 缓存放本地，推荐
2. 云函数（腾讯云 SCF）— 和微信小程序同生态，冷启动慢是问题
3. 本地 + 内网穿透 — 开发阶段临时用

## 五、核心交互流程

### 5.1 遥测对比

```
用户进入赛事详情
  → 切换到「遥测对比」Tab
  → 选择车手 A（下拉，显示车队颜色）
  → 选择车手 B
  → 点击「生成对比」
  → 前端请求后端 API
  → 后端用 FastF1 拉取最快圈遥测数据（优先从缓存读）
  → 返回 JSON（Distance, Speed, Throttle, Brake, Gear + 弯角信息）
  → 前端 ECharts 渲染 4 个联动子图
  → 横轴显示弯角编号（T1~T18）
  → 右上角显示两人圈时和差距
  → 底部如有数据丢包，显示灰色提示文字
  → 图表下方显示「AI 解读」入口按钮
```

### 5.2 圈时分析

```
用户进入赛事详情 → 正赛圈时分析 Tab
  → 默认展示所有车手圈时折线
  → 可筛选车手（多选）
  → 轮胎策略色块叠加在背景（软 / 中 / 硬 / 中间 / 全雨）
  → 点击某圈可查看该圈具体数据
  → 图表下方可触发 AI 稳定性 & 轮胎衰退分析
```

### 5.3 AI 分析报告（核心新增）

```
用户点击「AI 解读」按钮
  → 后端规则引擎先计算结构化指标（见下文）
  → 把指标数据拼成 prompt 发给 LLM API
  → LLM 返回自然语言分析报告
  → 前端 Markdown 组件渲染，分段展示
  → 用户可点击「收起 / 展开」各维度
```

## 六、AI 分析模块详细设计

### 6.1 分析维度

#### 维度一：弯道攻略对比
规则引擎按弯角分段，计算每个弯角：
- 刹车点距离差（m）
- 最低速差（km/h）
- 出弯速度差（km/h）

输出示例指标：
```json
{
  "corner": "T1",
  "brake_point_delta": "+8m (ALB brakes later)",
  "min_speed_delta": "-5 km/h (ALB slower at apex)",
  "exit_speed_delta": "+3 km/h (ALB faster exit)"
}
```

#### 维度二：赛段时间差分析
将全圈按赛道段（S1 / S2 / S3）拆分，计算各段时间差，定位哪段亏时最多。

输出示例指标：
```json
{
  "S1_delta": "+0.121s (ALO faster)",
  "S2_delta": "-0.089s (ALB faster)",
  "S3_delta": "+0.043s (ALO faster)"
}
```

#### 维度三：直线 & 油门效率
- 直线最高速对比
- 油门全开时间占比（全圈 & 各赛段）
- DRS 使用效果（开 / 关速度差）

#### 维度四：轮胎 & 稳定性
- 圈时标准差（越小越稳定）
- 每10圈圈时衰退斜率
- 轮胎换装时机与策略评价

### 6.2 Prompt 模板（规则 → LLM）

```
你是一名 F1 专业数据分析师，请根据以下结构化数据，生成一份专业但易读的中文分析报告。

【比赛信息】
赛事：{race_name}，{session_type}
对比车手：{driver_a}（{team_a}）vs {driver_b}（{team_b}）

【计算指标】
弯道攻略：{corner_data}
赛段时间差：{sector_data}
直线效率：{straight_data}
轮胎稳定性：{tyre_data}

【输出要求】
- 总体结论（2~3句，点出核心差距）
- 弯道攻略分析（结合具体弯角数据）
- 赛段优劣势分析
- 直线 & 油门效率
- 轮胎管理 & 稳定性
- 每个维度 100~150 字，用数据支撑结论
- 适合 F1 普通观众理解，专业术语需简短解释
```

### 6.3 数据流

```
FastF1 遥测数据
    │
    ▼
规则引擎（Python）
  ├── 弯角分段计算
  ├── 赛段时间差
  ├── 油门效率统计
  └── 轮胎衰退回归
    │
    ▼
结构化 JSON 指标
    │
    ▼
拼入 Prompt 模板
    │
    ▼
LLM API（Claude / GPT-4）
    │
    ▼
Markdown 文本报告
    │
    ▼
前端渲染展示
```

### 6.4 成本控制

- 规则引擎先算，只把精简后的结构化指标送给 LLM（不传原始遥测数组）
- 同一对比结果缓存 LLM 输出，避免重复调用
- 每次分析约消耗 1000~1500 tokens，成本极低

## 七、后端 API 设计

| 接口 | 方法 | 说明 |
|------|------|------|
| `/seasons` | GET | 返回可用赛季列表 |
| `/events?year=2026` | GET | 返回赛季赛历 |
| `/qualifying?year=2026&round=4` | GET | 排位赛结果 |
| `/telemetry?year=2026&round=4&d1=ALB&d2=ALO` | GET | 遥测对比数据 |
| `/laptimes?year=2026&round=4&session=R` | GET | 正赛圈时数据 |
| `/standings?year=2026` | GET | 积分榜 |
| `/analysis?year=2026&round=4&d1=ALB&d2=ALO&session=Q` | GET | AI 分析报告 |

返回格式统一：
```json
{
  "status": "ok",
  "data": { ... },
  "note": "ANT telemetry truncated at 5379m — F1 API packet loss"
}
```

AI 分析报告返回额外字段：
```json
{
  "status": "ok",
  "data": {
    "metrics": { ... },
    "report": "## 总体结论\n阿尔本在 T1-T3...",
    "cached": true
  }
}
```

## 八、数据质量处理原则

- 遥测截断：原样展示，`note` 字段说明原因，AI 报告中同步注明数据不完整
- 圈时 NaT：跳过不展示，不报错
- LLM 调用失败：降级展示规则引擎的纯数字结果，不影响图表
- 数据丢包提示统一用灰色小字，不影响主图阅读
