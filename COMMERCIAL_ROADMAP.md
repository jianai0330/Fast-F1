# F1 小程序商用级升级路线图

## 目标定位

本项目不应直接复制官方 F1 App / F1 TV 的实时数据产品路线，而应定位为：

> 中文 F1 数据理解与赛后复盘工具：把专业遥测、策略、积分、资讯和社区讨论翻译成普通车迷也能看懂、愿意分享、愿意反复打开的体验。

核心差异化：
- 中文语境：术语、策略、技术变更、新闻影响的中文解释。
- 赛后理解：用 FastF1 数据 + 规则引擎 + AI 生成复盘，而不是只堆图表。
- 轻量社区：围绕每条新闻、每场比赛、每个车队沉淀讨论。
- 低门槛可视化：先给结论，再给证据图，再给专业细节。

## 竞品基准

### 官方产品
- F1 App / F1 TV：强在 live timing、driver tracker、onboard、多视角、官方数据权威。
- 官方路线启发：实时状态、赛道位置、关键事件提示、订阅转化。

### 民间/第三方产品
- Formuletry / LumiLine / F101 / F1Dash / Pitwall：强在 live timing、telemetry、race control、strategy、replay、工程师视角。
- 第三方路线启发：专业数据面板、回放、策略模拟、个性化车手/车队关注。

### 本项目应避开的方向
- 不做“官方替代品”表达。
- 不把商用卖点建立在未经授权的 F1 官方商标、实时计时数据或版权内容上。
- 不追求第一阶段实时直播级体验，先打磨赛后/资讯/复盘闭环。

## 商用级体验框架

### 1. 首页：从赛历变成驾驶舱
- 下一站大卡片：倒计时、赛道、天气、轮胎策略、看点。
- 已结束比赛：复盘入口、关键事件、赢家输家、一键分享。
- 关注流：我关注的车手/车队新闻、积分变化、讨论热帖。

验收指标：
- 冷启动首屏可交互 < 1.5s（有缓存）。
- 首页 80% 内容来自缓存优先渲染，后台静默刷新。

### 2. 赛事页：从 tab 列表变成比赛故事
- 第一屏先给 3 个结论：胜负关键、策略转折、数据异常。
- 图表分层：摘要图 -> 关键圈/关键弯 -> 全量数据。
- 正赛图表统一 lazy init，避免 ec-canvas 条件渲染不稳定。

验收指标：
- 正赛页切 tab 无白屏、无 canvas node not found。
- 图表初始化失败率接近 0，可自动重试。

### 3. 遥测页：从全量曲线变成洞察工具
- 默认降采样显示完整圈，点击弯角后加载局部高清数据。
- 自动标注差异点：谁刹车更晚、谁出弯更快、谁 DRS/尾速占优。
- “普通解释/工程师解释”双模式切换。

验收指标：
- 遥测 tab 切换 < 200ms。
- 大数组不进入页面 `data`，仅保留私有缓存字段。

### 4. 资讯与 AI：从文章列表变成中文围场雷达
- 新闻按车队/术语/影响分类，而不是单纯时间流。
- AI 分析必须标注来源类型：原文事实、数据推算、观点判断。
- 分析结果可引用到论坛，形成讨论闭环。

验收指标：
- AI 分析可被重试、排队、限流。
- 每条分析有生成状态、失败原因、更新时间。

### 5. 社区：从论坛变成轻量车迷会
- 关注车手/车队后生成个性化讨论流。
- 帖子支持举报、审核、删除留痕。
- 热帖排序按分区、时间衰减、互动质量综合。

验收指标：
- 用户身份不再信任客户端传 openid。
- 发帖/评论/AI 触发都有频控。

## 流畅度优化框架

### 前端 P0
- 遥测和圈时大对象移出 `setData`，改为 `this._telemetryCache`、`this._laptimesCache`。
- event / standings 图表统一改成 `lazyLoad: true + selectComponent().init()`。
- `onShow` 刷新改成脏标记机制：只有发帖、投稿、分析状态变化时刷新。
- 新闻列表点击只传 `id`，不要 `data-item="{{item}}"` 挂整对象。

### 前端 P1
- 封装统一图表容器组件：loading、empty、error、cached、retry、dispose、resize。
- 图表数据降采样：默认 1/3 或 1/5 点数，局部查看再用高清数据。
- 术语热词延迟加载：不阻塞首屏和图表初始化。

### 后端 P0
- 重接口加 page_size 上限。
- FastF1 session cache 增加 TTL、锁、容量限制。
- AI 分析、精选分析、爬虫、预热全部进入队列，不再在 API 请求里随手起 daemon thread。

## 商用级工程框架

### 生产治理 P0
- 安全：移除默认 admin token；限制 CORS；登录态签名/JWT；接口频控。
- 可观测性：请求耗时日志、trace id、慢接口日志、任务状态表、`/healthz`、`/readyz`。
- 部署：`deploy.sh` 增加 `set -euo pipefail`、部署前验证、健康检查、失败回滚。
- 任务：APScheduler 从 Web 进程拆出，定时爬虫和 AI 任务由 worker 执行。

### 数据与合规 P1
- 数据库：SQLite 短期加 busy_timeout、备份校验、迁移记录；中期迁 PostgreSQL + Alembic。
- 用户数据：隐私政策、用户协议、数据删除、内容举报、审核留痕。
- 第三方内容：新闻摘要/引用比例控制、来源链接、版权声明。
- F1 合规：避免官方关联表述；商用前确认 Formula 1 商标、计时/结果数据、AI 使用授权边界。

## 阶段计划

### Phase 0：商用底座止血（1 周）
- 清理 git 运行态文件：`.env`、db、cache 不进仓库。
- 移除前端硬编码 admin token，后端强制 ADMIN_TOKEN env。
- 增加 health/readiness、请求耗时日志、慢接口日志。
- `deploy.sh` 增加部署前 compile、健康检查、失败提示。

### Phase 1：流畅度专项（1-2 周）
- 大数组移出 `setData`。
- event / standings 图表 lazy init。
- 新闻/论坛 onShow 脏标记刷新。
- 遥测数据降采样。

### Phase 2：核心产品升级（2-4 周）
- 首页改驾驶舱。
- 赛事页新增“比赛故事/关键转折”首屏。
- 遥测页新增差异点自动标注。
- 新闻页新增车队/术语/影响筛选。

### Phase 3：AI 复盘产品化（3-5 周）
- 新建 race review：关键时间线、策略转折、赢家输家、数据证据。
- AI 任务队列化，支持状态、重试、失败原因。
- 复盘可分享成卡片。

### Phase 4：商用运营闭环（持续）
- 用户关注车手/车队。
- 轻量会员权益：高级复盘、双车对比、历史赛季、导出图卡。
- 内容审核、举报、运营后台。
- 合规授权路径评估。

## 北极星指标

- 首屏可交互时间：缓存命中 < 1.5s，冷启动 < 3s。
- 图表稳定性：关键页 canvas 初始化失败率 < 0.5%。
- 复访理由：用户关注对象后，每次打开都有“新看点”。
- 内容质量：每场比赛至少 1 篇可分享复盘，AI 分析事实错误率持续下降。
- 运营安全：所有后台任务可追踪、可重试、可限流、可回滚。
